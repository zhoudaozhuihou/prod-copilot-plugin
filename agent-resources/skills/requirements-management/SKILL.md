# Requirements Management Skill

Use this skill for requirement discovery, clarification, mapping, prioritization, review, and traceability.

## Workflow
1. Capture raw demand without prematurely designing the solution.
2. Normalize each demand into a stable requirement ID.
3. Separate facts, assumptions, open questions, constraints, and risks.
4. Ask only clarification questions that change scope, cost, risk, compliance, acceptance criteria, or implementation sequencing.
5. Map Business Goal → User Need → Capability → Feature → User Story → Acceptance Criteria → Task/Test.
6. Prioritize with RICE, MoSCoW, Kano, dependency order, compliance urgency, and delivery risk.
7. Create traceability to PRD, UI, API, backend, data, SQL, tests, release, and runbook artifacts.

## Guardrails
- Do not invent business rules, data fields, legal constraints, or owners.
- Do not jump to code before acceptance criteria and scope are clear.
- Use evidence labels: User Input, Attachment, Repo Evidence, Policy Pack, Assumption.
- In banking/data contexts, always consider permissions, privacy, audit, DQ, reconciliation, lineage, SLA, release and runbook implications.

## Output Quality Bar
Every requirement should be understandable, testable, traceable, prioritizable, and reviewable.
